# 批判性分析：gstack 设计哲学报告的价值评估 (Critical Analysis: Value Assessment of gstack Design Philosophy Report)

> **评估对象：** `docs/reports/2026-03-30-design-philosophy-extraction-from-garrytan-gstack`
> **评估方法：** 第一性原理 + 事实核查
> **日期：** 2026-03-30

---

## 一、报告的事实准确性核查 (Fact-Check)

报告中有几个声称与 loom 实际状态不符：

| 报告声称 | 实际情况 | 严重程度 |
|----------|----------|----------|
| loom 有 19 个技能 | 实际 20 个（agent-browser 未计入？） | 低（数字不影响结论） |
| 图表规范在多个 SKILL.md 中内联重复 ~250 行 | `docs/conventions/terminal-diagram-style.md` 已存在，但 5 个 design 技能仍有各自专用的图表明细（不完全是重复） | **中**（报告将"各技能有专用图表规则"等同于"重复"，这是过度简化） |
| loom 缺少"完整性优先"原则 | design-readiness-check 本质就是完整性门禁，只是没有贴"Boil the Lake"的哲学标签 | 低（标签 vs 实质） |

## 二、逐条价值评估 (Finding-by-Finding Assessment)

### 高价值发现

#### 1. Fix-First 评审模式（AUTO-FIX / ASK 分类）⭐⭐⭐⭐⭐

**为什么有价值：** 这是一个可立即验证的行为改变，不是架构改动。当前 loom 的所有检查类技能（design-readiness-check、design-decision-audit）只做"report"——列出问题让用户自己修。但其中很多问题是机械性的（如 missing status marker、未填写的决策字段），完全可以让 AI 直接修复。

**第一性原理验证：** 用户时间 > AI 时间。如果修复一个问题只需要 5 秒 AI 时间但需要 30 秒用户阅读判断时间，自动修复就是正确的默认行为。

**实施建议：** 在 design-readiness-check 的 4 个 subagent 中，让 assumption-checker 和 risk-checker 输出时区分 `AUTO-FIX` 和 `ASK` 两类。

#### 2. 用户主权循环：`AI 生成 → 用户验证 → AI 不得跳过验证` ⭐⭐⭐⭐

**为什么有价值：** 这是一条简练的设计原则。loom 的 design 系列技能虽然隐含了这个模式（readiness-check 就是验证门），但没有将其显式化。当技能越写越复杂时，缺少显式原则会导致"验证步骤可选化"的退化。

**第一性原理验证：** AI 最大的风险不是能力不足，而是过度自信。强制验证循环是针对这个根因的设计。

#### 3. Review Status Dashboard ⭐⭐⭐⭐

**为什么有价值：** design-orchestrator 做路由决策时，缺乏对"当前状态完成度"的可视化。用户看到的是一个技能名，而不是"你完成了 3/5 个设计阶段"。Dashboard 提供了决策上下文。

**但需注意：** gstack 的 Dashboard 是给 `/ship` 用的（发布前检查），loom 的等价场景是 design-readiness-check（设计完成前检查）。直接套用场景不同，但模式通用。

### 中等价值发现

#### 4. 三层知识模型（Layer 1/2/3） ⭐⭐⭐

**为什么有价值：** 提供了一个决策框架。但报告自己承认 loom 已覆盖了 Layer 2（`[RESEARCH]` 标记），而 Layer 1（成熟模式直接用）和 Layer 3（第一性原理）是所有工程师的默认行为——不需要一个标签系统来提醒。

**批判：** 这个模型看似清晰，但实操中三层之间的边界模糊。什么时候一个模式算"经过验证"？什么时候算"新兴热门"？分类本身就需要判断力，而分类标准的缺失使得这个框架更像是一个思考练习而非可执行工具。

#### 5. 在技能中注入 memory 搜索 ⭐⭐⭐

**为什么有价值：** 当前 loom 的 auto memory 系统是被动存在的——Claude 会读 memory 但技能不会主动搜索。在 design-readiness-check 执行前自动搜索相关的 project/feedback memory 可以避免重复犯错。

**但需注意：** Claude 的 auto memory 系统本身已经在每次对话开始时加载 MEMORY.md。额外的"主动搜索"是否真的带来增量价值？取决于 memory 的数量和相关性。如果只有 5-10 条 memory，全部加载已经足够。

#### 6. 轻量共享上下文块 ⭐⭐⭐

**为什么有价值：** design 系列技能确实共享一些概念（design_state、设计原则），但没有一个统一的地方定义它们。当这些概念演变时，需要手动更新多个 SKILL.md。

**批判：** 报告建议"用文件引用（`@conventions/...`）做轻量去重"。这实际上是 Claude Code 的 `@` 引用机制。可行性取决于 Claude Code 是否支持在 SKILL.md 中使用 `@` 引用——如果不支持，这个建议就是空谈。

### 低价值 / 过誉的发现

#### 7. "Boil the Lake" 哲学 ⭐⭐

**批判：** 这个原则听起来深刻，但实际含义是"能做完的就做完"——这是一个不需要命名的常识。报告把它包装成 gstack 的核心哲学，但它更像是 gstack 作者对自身行为的事后合理化。

**loom 是否需要这个原则？** 不需要。design-readiness-check 的存在已经证明 loom 在实践"完整性优先"，只是没有贴标签。添加一个哲学标签不会改变行为。

#### 8. 模板生成系统 ⭐⭐

**批判：** 报告自己也承认"loom 的技能规模（19 个）可能还不值得引入完整的模板引擎"。那为什么把它列为"长期考虑"？一个需要规模扩大到 30+ 才值得的建议，对当前阶段的决策没有影响。

**更深层的问题：** gstack 的模板系统是为了解决 31 个技能中大量重复 preamble 的问题。但 loom 的技能分工更清晰（design 阶段专精），重复内容更少。这是两个不同的架构选择导致的结果，不是一个"落后 vs 先进"的比较。

#### 9. 跨模型评审（Claude + Codex） ⭐⭐

**批判：** 报告自己标注了"优先级不高"。这更像是 gstack 的一个技术展示而非核心设计理念。双模型评审的前提是两个模型有足够不同的错误分布——但当前 Claude 和 GPT 的常见错误模式正在趋同。

**实际价值：** 在高风险决策（如架构选型）时引入第二意见有价值，但这不需要一个专用技能。只需要在 decision-evaluation 的 prompt 中加一句"如果你不确定，建议用户寻求第二意见"。

#### 10. 浏览器守护进程 ⭐

**批判：** 报告正确地指出"loom 不需要复刻"。那么这个发现在报告中的价值是什么？作为"gstack 的差异化壁垒"的说明可以，但作为"可供 loom 借鉴的模式"则没有价值。报告将它放在"架构模式"章节，给人一种可以学习的暗示，但实际上只是信息展示。

#### 11. Preamble 分层系统 ⭐⭐

**批判：** 4 级分层是 gstack 为解决自身问题（31 个技能、不同复杂度）设计的。loom 只有 20 个技能且定位专精，分 2 类（操作型 vs 设计型）的报告建议是对的，但这个结论不需要研究 gstack 的 4 级分层就能得出——这是简单的工程判断。

## 三、报告的盲点 (What the Report Misses)

### 1. 忽略了 loom 的核心优势：显式状态管理

报告对比表中提到 loom 用"显式 design_state 合约"，但后续分析中完全没有展开这个优势。gstack 的状态管理是"文件传递（隐式）"——技能之间通过文件系统传递状态，没有统一的状态模型。

loom 的 `design_state` 是一个更好的设计：它让设计过程的状态可查询、可验证、可路由。这个优势被报告一笔带过了。

### 2. 没有分析 gstack 的弱点

报告是一份"学习笔记"，不是"批判性评估"。它只看 gstack 做了什么好，没看 gstack 做了什么不好：
- 31 个技能的认知负担（用户如何知道该用哪个？）
- 隐式状态传递的可靠性问题（如果中间一步的文件格式变了怎么办？）
- 模板生成系统的维护成本（改 preamble 需要重新生成所有技能）

### 3. 没有考虑采纳成本

报告列出了 7 个 actionable takeaways，但没有评估每个建议的采纳成本。比如"为 design-orchestrator 添加 Review Status Dashboard"——这需要改动 design-orchestrator 的路由逻辑、增加状态追踪、定义仪表板格式。这不是一个"Quick Win"，而是一个需要设计的中型任务。

## 四、筛选后的可信建议 (Filtered Recommendations)

按实际价值和实施成本排序：

### 应该做（高价值 + 可行）

1. **在检查类技能中引入 AUTO-FIX / ASK 分类** — 在 design-readiness-check 的 subagent 输出中区分可自动修复和需人工判断的问题。实施成本低（只改 subagent prompt），价值高（减少用户来回次数）。

2. **显式化"AI 生成 → 用户验证"原则** — 在 design-orchestrator 或 design-readiness-check 的 SKILL.md 中增加一行原则声明，防止未来技能退化。成本几乎为零。

### 可以考虑（中等价值 + 中等成本）

3. **为 design-orchestrator 增加状态摘要** — 不是 gstack 那样的完整 Dashboard，而是在路由时显示"当前 design_state 各字段的完成状态"。成本中等，价值中等。

4. **验证 `@` 引用机制是否可用于图表规范去重** — 如果 Claude Code 支持在 SKILL.md 中使用 `@docs/conventions/...`，则可以将 5 个 design 技能的内联图表规则替换为引用。需要先做技术验证。

### 不应该做（低价值或成本不匹配）

5. ~~模板生成系统~~ — 当前规模不需要
6. ~~全生命周期扩展~~ — 与 loom 的"设计阶段专精"定位矛盾
7. ~~跨模型评审~~ — 投入产出比不合理
8. ~~Preamble 分层~~ — 过度工程化

---

## 五、总结 (Conclusion)

报告的核心价值在于 **Fix-First 评审模式** 和 **用户主权循环** 两个发现。其余大部分内容是信息性的——有助于理解 gstack 的设计选择，但不构成 loom 需要采纳的建议。

报告的主要问题是**缺乏批判性视角**：它是一份好的"学习笔记"，但不是一份好的"决策依据"。它没有区分"gstack 做得好是因为它解决了一个 loom 没有的问题"和"gstack 做得好是因为它有一个通用性更好的方案"。

**一句话总结：** 从 7 个建议中，真正值得实施的只有 1-2 个。报告最有价值的部分不是建议，而是它暴露的设计维度（AUTO-FIX/ASK 分类、验证循环显式化）——这些维度 loom 之前没有命名。
